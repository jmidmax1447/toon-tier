## Transcript Highlights

## Highlight 1 Initial Setup
- The first highlight would be the first two prompts I used to begin developing the application after the planning phase. The first prompt used the contents of my README.md to summarize the project, as well as the features and technologies I wanted to use. Providing the AI with this context made it much more efficient than having it search through the existing code to piece together what I was trying to build. The second prompt then used this context to help create a plan for micro-iteration, building specific features in small steps so individual parts could be tested before continuing the build. This approach helped define a clear development plan and made debugging a much easier process.

## Highlight 3 .ENV file creation
- When implementing the .env file to store my database URL and API key, I made sure that the values were left empty in the code and were not passed directly through the AI agent in the CLI interface. Instead, the sensitive values were added locally after the configuration was generated. This part of the conversation was important because it highlights good security practices when working with AI-assisted development. Passing credentials directly through the CLI interface could expose sensitive information, so separating configuration and doing it manually seemed like the best choice.

## Highlight 3 Database Tables and when they're made
- The third highlight takes place around step 4–5 in the initial micro-iteration plan, when certain components that needed data from the database were being developed. It occurred to me that we had not created any tables in Supabase yet, so I prompted the AI to think about when we should introduce the necessary tables while developing the components. I preferred creating them alongside the components so I could see the display and the database connection working at the same time. Additionally, since the AI could not have direct access to the technology, I had it generate the queries that I could input into the SQL editor.

## Highlight 4 Complete Tier feature integration and iterating
- After the Base Tier site was complete, I decided to begin working on the bonus features for the Complete Tier and started with customizable profile pages. When prompting the development of these features, specifically the image input, it initially started as a link input, so I told the AI to reiterate and change it to an image upload with the provided context of where it would be stored on Supabase. However, when making changes to the base code that was already established, GitHub Copilot would occasionally leave extra curly braces or lines of code from previous functions. To fix these errors, I found that copying the React error messages and relaying them back to Copilot was the quickest and most efficient solution, since it could identify exactly what the issue was and which line in the code the problem existed on.

## Highlight 5 Populating the database
- Once the site was completed, it needed more users and entries in their respective tables to make it more presentable. Doing this manually would have taken much longer, so I decided to grab the UIDs of the users I had manually created and prompted the AI to generate SQL queries to fill in believable test data that could be used on the site. As a result, I quickly obtained around 10 accounts, each with 10–30 reviews spread across them, to simulate what the site would look like with multiple users actively using it.
